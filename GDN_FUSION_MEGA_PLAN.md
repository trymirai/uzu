## PART 1 - Fusions F1-F4 (target: 10 -> 6 dispatches)

Apply in this order; keep correctness tests green after each step (update test
harness wiring as kernels merge - the pipeline test drives the kernel chain).

F1 - Fold DecayScale into FusedApply. decay_scale[t] = beta_t * exp(g_last - g_t)
is a per-token scalar. Give the fused kernel `beta` as an input; compute the
scale on the fly in the update phase (it already reads g there). Delete the
DeltaNetChunkedStateA2DecayScale kernel + its CPU mirror + the decay_scale
buffer. chunked_apply.metal / chunked_apply.rs then become empty - delete both
files (and the cpu mod.rs entry).

F2 - Merge BuildW + BuildU -> one DeltaNetChunkedBuildWU kernel. The two are
structurally identical (same a_packed/a_inv fragment loads, same block forward
substitution; different RHS: scaled k_norm vs beta*V) - one kernel emitting both
outputs shares all A-matrix loads. Saves a dispatch at small T and should cut
the isolated 0.47+0.64ms at T=4096 to ~0.7-0.8ms. NOTE: Part 2 Mode L deletes
BuildWU entirely; still do F2 (it is cheap, and remains the fallback path if
Mode L underperforms at large T), but do not over-polish it.

F3 - Merge ScaleQk into Gram. Gram computes kk and qk per K-HEAD; ScaleQk
expands qk to per-V-HEAD applying causal mask + exp(g_row - g_col). Fuse: the
gram kernel, after its MMA, writes qk_scaled for each of the 3 v-heads of its
k-head directly (per-v-head g), keeping the kk output for Solve. Deletes the
ScaleQk dispatch + the intermediate qk buffer. Grid stays per k-head.

F4 - Delete the Cumsum kernel + the g buffer. Consumers recompute the
chunk-local prefix from log_decay (64 adds/chunk-head - noise):
- Solve: switch to its existing RECOMPUTE_G=true variant.
- GramScaled (F3): compute g inline from log_decay.
- FusedApply: once per chunk, compute the prefix into a small TG array
  (64 floats + g_last; ~few hundred bytes on top of the existing 24KB), use for
  the Y-phase exp(g) scale, alpha=exp(g at last VALID token), and the F1 decay
  scale. Mind the partial-last-chunk clamp.
The fused kernel's inputs become: w, u, q_norm, k_norm, qk_scaled, log_decay,
beta, state, out.

End state Part 1 pipeline — AS BUILT (7 dispatches, NOT 6):
prep -> cumsum -> gramScaled -> solve -> buildWU -> fusedApply -> normGate.

**F4 WAS REVERTED (critical lesson for Phase B).** RECOMPUTE_G is O(C^3): the
chunk-local cumsum recompute (`chunked_g`, O(C)) sits inside solve's O(C^2)
substitution loops, blowing solve from ~0.33ms to 6.29ms at T=4096. So Cumsum
stays as its own kernel and ALL consumers (solve, gram, apply, and Phase-B
megaApply) read the PRECOMPUTED `g` buffer. NEVER dispatch a RECOMPUTE_G=true
variant in the shipping pipeline, and NEVER do a per-chunk single-thread g
prefix scan inside the scan kernel — read g from the buffer.

F1/F2/F3 were kept and are net-positive: merged BuildWU (0.77ms < old
build_w+build_u 1.10ms), merged Gram+ScaleQk (0.65ms < old 0.71ms), DecayScale
folded into apply (apply rose ~0.47ms but the net 7-dispatch pipeline is faster
than both recurrent and the old fused-10 everywhere at T>=256; crossover moved
from ~512 to ~256). Phase-A checkpoint commit: c2b01bd6.

Phase-A measured (M5 Max, median ms, recurrent / fused-7): T=64 0.406/0.488,

## PART 2 - Mega kernel ("chunked prefill style")

Enabling identity (exact, from W = T.diag(beta*e^g).K and U = T.diag(beta).V,
where T = the unit-lower-triangular inverse the solve produces):

    Vnew = T . R,   R = diag(beta) . (V - e^g (.) (K . S0^T))

K.S0^T has the same shape/cost as the current W.S0^T and uses the state slice
the scan TG already holds. So the scan can compute R itself and apply T as one
dense [64,64]x[64,VT] MMA - W and U (and BuildWU, and their buffers) disappear.
This is FLA's `fused_chunk` mode; we add an Apple-specific twist (Mode S).

New kernel `DeltaNetChunkedMegaApply` (new file chunked_mega.metal + CPU mirror
chunked_mega.rs), same skeleton as chunked_fused.metal (TG per (head,v-slice),
S^T f32 16KB in TG, VT variants). Keep chunked_fused.metal intact until the
final decision gate - do not delete the incumbent while experimenting.

Mode L (all T) - per chunk phases:
1. R phase: R = beta*(V - e^g (.) (K.S0^T)) into the 8KB TG buffer. V is read
   directly from in_proj (strided rows, exactly as BuildU did); K from k_norm;
   scalars from log_decay/beta (F4-style inline prefix). Barrier.
2. Vnew phase: Vnew = T.R computed into register tiles (T streamed from device),
   barrier, store Vnew over the R buffer.
3. Y phase: Y = e^g (.) (Q.S0^T) + A.Vnew -> out. (A = qk_scaled, streamed.)
4. Update phase: S <- alpha*S + Vnew^T.B (B = decay-scaled k_norm, on the fly).
~4 barriers/chunk. TG budget unchanged: 16KB S^T + 8KB R/Vnew.

Solve change: emit dense T [C,C] per (chunk, v-head) - one extra substitution
pass with identity RHS over the existing block inverses (~0.26 MFLOP/chunk-head,
chunk-parallel). Store T as BF16 ([chunks,HV,64,64] ~25MB at T=4096; replaces
the bf16 W+U buffers, so numerics are no worse than the old path - state stays
f32). Keep block a_inv/a_packed outputs while BuildWU still exists.

Mode L pipeline — AS REVISED (6 dispatches, NOT 5): prep -> cumsum ->
gramScaled -> solveT -> megaApply -> normGate. (Cumsum stays per the F4 lesson
above; megaApply and gram read the precomputed g buffer. solveT can emit dense
T alongside gram's qk_scaled, both consuming g.)

Mode L risk (be honest in reports): scan critical path grows (~45% more MMA + 1
barrier/chunk) but the scan is latency-bound, so much may hide; W/U traffic
(~350MB at T=4096) is eliminated. Whether T=4096 nets out better or worse than
the Part-1 pipeline is THE question - measure, don't argue.

PHASE B RESULT (Mode L): ADOPTED. 7 dispatches (Solve retained; SolveT is a
separate DeltaNetChunkedSolveT that does block-MMA over Solve's a_packed/a_inv,
NOT a self-contained solve — the naive self-contained O(C^3) version was a 7x
perf trap, same lesson as F4). megaApply (chunked_mega.metal, 5 barriers/chunk,
24KB TG) beats fused-7 at every T (1.03-1.13x), beats recurrent at every T>=128.
M5 Max medians (recurrent / Mode-L VT32): T=64 0.407/0.473, 128 0.400/0.415,
256 0.461/0.433, 512 0.792/0.693, 1024 1.351/1.129, 4096 4.949/3.926,
32768 42.04/28.81. Breakdown T=4096: megaApply in-context 2.31ms, precompute
1.12ms (solve 0.34 + solveT 0.29 replaces solve+buildWU 1.09). Recurrent still
wins at T=64 only. For Mode S, the in-TG solve MUST use block 16x16 inversions
(like chunked_solve), never naive per-column O(C^3) substitution.

Mode S (small T, the T=64 play) - same kernel, template/function-constant
INLINE_PRECOMPUTE: each TG additionally computes gram + solve + A inline in TG
memory for each chunk instead of reading qk_scaled/T from device (redundant
x4 slice-TGs per head and x3 v-heads per k-head - irrelevant at small T, which
is dispatch-bound). Pipeline collapses to 3 dispatches: prep -> megaApply(S) ->
normGate - dispatch parity with the recurrent path, with MMA math instead of
recurrent's 64 serial simd_sum steps.

Mode S TG-memory budget is the key design constraint (32KB cap): S^T f32 16KB
is fixed (state precision non-negotiable). The kk/qk/A/T scratch must fit the
remaining ~16KB via some mix of: bf16 scratch for kk/T/A (acceptable - the old
pipeline's W/U were bf16; state stays f32), in-place buffer rotation
(kk -> A_solve -> T in one buffer; qk -> A masked in place), or VT=16 for
Mode S (S^T drops to 8KB; grid 384 TGs). Implementer's choice - document it.
The in-TG solve is the hard part: block 16x16 inversions (serial-ish, one or
two simdgroups) while others do gram, then barrier; follow chunked_solve.metal's
math. If Mode S proves intractable within budget, report why and stop Mode S -
Mode L + Part 1 still stand.
